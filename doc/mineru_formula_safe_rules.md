# MinerU 公式安全修正ルール

本ルールは、MinerU / UniMERNet 系の公式 OCR 出力である `raw latex-ish` テキストのみを対象とします。

目的は「すべての公式を最も美しく直す」ことではなく、次のとおりです。

- 低リスクで構造化され、説明可能な修正のみ行う
- 複雑な公式を壊すグローバル文字列置換を避ける
- Typst / mitex レンダリングをより安定させる

## 背景

MinerU の公式出力は、本質的に公式認識モデルが生成した LaTeX 列であり、PDF ネイティブの正規 LaTeX ではありません。

よくある特徴:

- 制御語と波括弧の間に空白: `\mathrm { C H }`
- 下付き・上付きが空白で分断: `x _ { i , j }`
- 数字の途中に空白: `1 . 2 7`
- 局所スタイルコマンドの混入: `\bf { g }`

## 許可する修正

次のルールはリスクが低く意味が明確なため残してよいです。

- `\mathrm { C H } -> \mathrm{CH}`
- `a _ { b } -> a_{b}`
- `a ^ { b } -> a^{b}`
- `a _ 2 -> a_2`
- `a ^ 2 -> a^2`
- 数字内部の空白を詰める: `1 . 2 7 -> 1.27`
- `\textsuperscript{...} -> ^{...}`
- 少数の明確な OCR ノイズ修正、例: `\mathrm { e V } -> \mathrm{eV}`

## 禁止する修正

次のルールはグローバル文字列置換で使ってはいけません。

- グローバル `"{ -> ("`、`"} -> )"`
- グローバル `" _ {" -> "_("`
- グローバル `" ^ {" -> "^("`
- 正規表現による大規模な波括弧削除
- 構文境界を確認せず `\frac`、`\sqrt`、`\left...\right` を直接書き換える
- 長さや文字集合から複雑な公式構造を「推測」する

## 複雑構造の扱い原則

次の構造は、構造化された平衡括弧レベルの処理のみ許可します。

- `\frac{...}{...}`
- `\sqrt{...}`
- `\left ... \right`
- `x_{i,j}`
- `E_{g}^{dir}`
- `\Delta G_{H^*}`
- `\mathbf{v}_{t+1}`

原則:

- まず `_` / `^` の原子結合を詰める
- 次にグループ内を再帰処理
- 未知のグループに対して積極的な書き換えはしない

## エンジニアリング上の境界

次の事実を受け入れます。

- MinerU 出力は最終 LaTeX ではない
- 複雑な公式は過剰修正より少修正を優先
- 複雑構造が既にレンダ可能なら原状維持を優先
- レンダ失敗時は selective sanitize を優先し、メインパスで積極的書き換えしない

## 現在の方針

現在のコードは次に従うこと。

- `normalizer.py` はホワイトリスト化された低リスクな詰めのみ
- `typst_formula_renderer.py` はグローバル波括弧置換をしない
- 複雑なスクリプト構造は平衡括弧の再帰処理で扱う
- 新ルールは必ず回帰テストを追加してから本番投入
