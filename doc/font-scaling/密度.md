# 密度はどう判定しているか

テキストボックスを紙の箱と想像すると、「密度」が答えているのは次の一文です。

**この翻訳結果を、現在のフォントサイズと行送りのまま箱に入れたとき、ゆったりか、ちょうどか、それとも過負荷か。**

直感では分かりやすい一方、コードに落とすときは単一指標だけでは足りません。「詰まっている」には少なくとも二層の意味があります。

- コンテンツ自体が長くなったか
- その箱がそのコンテンツを受け止められるか

したがって現行実装では、密度は単一の定数ではなく、複数の関数が連携して判定します。

---

## 一、結論：実際には二種類の密度を見ている

現行実装で「密度」に最も関係するのは単一関数ではなく、次の二系統です。

1. **長さ密度**
   - 翻訳後のコンテンツが原文より「膨張」したか
   - 対応関数：`translation_density_ratio(...)`

2. **レイアウト密度**
   - 現在のフォントサイズと行送りで、箱の中が詰まりすぎないか
   - 対応関数：`layout_density_ratio(...)`

[fit.py](backend/scripts/services/rendering/layout/payload/fit.py) では、これら二指標がともに判定に使われます。

```python
length_density_ratio = translation_density_ratio(item, protected_text)
layout_density = layout_density_ratio(box, protected_text, font_size_pt=font_size_pt, line_step_pt=line_step)
is_dense_block = length_density_ratio >= COMPACT_TRIGGER_RATIO or layout_density >= LAYOUT_COMPACT_TRIGGER_RATIO
```

つまりシステムは「翻訳が長くなったか」だけでも、「箱が満杯に近いか」だけでもなく、**両方**を問います。

---

## 二、第一層：コンテンツが明らかに膨張したか

最も素直な考え方は、原文は短いのに翻訳が長い段落になれば、組版は難しくなる、ということです。

この層は [text_common.py](backend/scripts/services/rendering/layout/payload/text_common.py) の `translation_density_ratio(...)` が担当します。

```python
def translation_density_ratio(item: dict, protected_text: str) -> float:
    source_words = source_word_count(item)
    if source_words <= 0:
        return 0.0
    zh_chars = translated_zh_char_count(protected_text)
    if zh_chars <= 0:
        return 0.0
    return zh_chars / source_words
```

この関数の処理は単純です。

- 原文の英単語数を数える：`source_word_count(item)`
- 翻訳後の中国語文字数を数える：`translated_zh_char_count(protected_text)`
- 「中国語文字数 / 原文語数」で比率を得る

意味は「精密な組版計算」ではなく、次を素早く見極めることです。

**この翻訳は、視覚的に詰まりやすくなったか。**

### 例

ある原文ブロックを仮定します。

- 原文語数：20
- 翻訳後中国語文字：18

このとき：

`translation_density_ratio = 18 / 20 = 0.9`

境界付近のややタイトな状態です。

別のブロックでは：

- 原文語数：20
- 翻訳後中国語文字：24

このとき：

`translation_density_ratio = 24 / 20 = 1.2`

これは「明らかな膨張」ブロックで、以降はより保守的に扱います。

閾値も同ファイルに定義されています。

- `COMPACT_TRIGGER_RATIO = 0.9`
- `HEAVY_COMPACT_RATIO = 1.0`

平易に言うと：

- `>= 0.9`：タイトになり始めた
- `>= 1.0`：重度のコンパクトブロック

## 三、第二層：現在のフォントサイズで箱が本当に埋まるか

第一層は「コンテンツが長くなったか」だけで、箱は見ていません。

同じ `ratio = 1.0` の二段落でも：

- 幅 400pt の本文枠なら問題ないことが多い
- 幅 160pt のキャプション枠ならすぐあふれる

第二層では「箱の容量」が必須です。

これは [text_common.py](backend/scripts/services/rendering/layout/payload/text_common.py) の `layout_density_ratio(...)` が担当します。

```python
def layout_density_ratio(
    inner: list[float],
    protected_text: str,
    *,
    font_size_pt: float,
    line_step_pt: float,
) -> float:
    width = max(8.0, inner[2] - inner[0])
    height = max(8.0, inner[3] - inner[1])
    zh_chars = translated_zh_char_count(protected_text)
    approx_char_width = max(font_size_pt * 0.92, 1.0)
    chars_per_line = max(4.0, width / approx_char_width)
    required_lines = max(1.0, zh_chars / chars_per_line)
    occupied_height = required_lines * line_step_pt
    return occupied_height / height
```

ロジックを言葉にすると次のとおりです。

1. 箱の幅・高さを見る
2. 現在のフォントサイズで一文字の概算幅を置く
3. 一行あたりの概算文字数を出す
4. 翻訳テキストの概算行数を出す
5. その行が占める高さを出す
6. 占有高さを箱の高さで割る

結果の比は直感的です。

- `< 1.0`：理論上は収まる
- `≈ 1.0`：非常にタイト
- `> 1.0`：理論上ははみ出し

### 例

ある箱を仮定します。

- 幅：180pt
- 高さ：90pt
- 現在のフォントサイズ：9pt
- 現在の行送り：12pt
- 翻訳後中国語文字：72

概算：

- 一文字幅 ≈ `9 × 0.92 = 8.28pt`
- 一行 ≈ `180 / 8.28 ≈ 21.7` 文字
- 72 文字 ≈ `72 / 21.7 ≈ 3.3` 行
- 占有高さ ≈ `3.3 × 12 = 39.6pt`
- レイアウト密度 ≈ `39.6 / 90 = 0.44`

このブロックは詰まっていません。

同じ内容を別の箱に入れると：

- 幅：110pt
- 高さ：48pt

このとき：

- 一行 ≈ `110 / 8.28 ≈ 13.3` 文字
- 72 文字 ≈ `72 / 13.3 ≈ 5.4` 行
- 占有高さ ≈ `5.4 × 12 = 64.8pt`
- レイアウト密度 ≈ `64.8 / 48 = 1.35`

典型的高密度ブロックで、現在のフォントサイズは大きすぎます。

## 四、第三層：箱の「実容量」はどう計算するか

上記 `layout_density_ratio(...)` は軽量な概算で、密度判定の前段向けです。

「この箱に実際どれだけ入るか」に近い計算は [capacity.py](backend/scripts/services/rendering/layout/payload/capacity.py) にあります。

中核は `box_capacity_units(...)` です。

```python
def box_capacity_units(
    inner: list[float],
    font_size_pt: float,
    leading_em: float,
    visual_lines: int | None = None,
) -> float:
    width = max(8.0, inner[2] - inner[0])
    height = max(8.0, inner[3] - inner[1])
    line_step = max(font_size_pt * 1.02, font_size_pt * (1.0 + leading_em))
    lines = max(1, int(height / line_step))
    if visual_lines and visual_lines > 1:
        lines = min(lines, max(1, visual_lines + 1))
    chars_per_line = max(4.0, width / max(font_size_pt * 0.92, 1.0))
    return lines * chars_per_line * 0.98
```

三つのことを行います。

1. フォントサイズと行送りから入る行数を推す
2. 箱幅から一行の容量を推す
3. 両者を掛けて総容量を得る

重要なのは `visual_lines` です。

幾何だけで「高さに何行入るか」を信じ切らず、OCR / 版面構造から得た視覚行数も参照し、容量を楽観しすぎないようにしています。

---

## 五、第四層：需要は単純な文字数ではない

容量が分かっても「需要」はどう測るか。

同ファイルの `text_demand_units(...)` が担当します。

```python
def text_demand_units(protected_text: str, formula_map: list[dict]) -> float:
    formula_lookup = {entry["placeholder"]: entry["formula_text"] for entry in formula_map}
    return sum(token_units(token, formula_lookup) for token in tokenize_protected_text(protected_text))
```

意味は次のとおりです。

- テキストをトークンに分割
- 通常文字は通常単位
- 数式プレースホルダは 1 文字扱いにせず、視覚コストに近い重みで計上

文字数だけだと数式の圧力を過小評価します。

### 例

次の二文は文字数が近いことがあります。

1. `该方法显著提高了材料性能。`
2. `该方法显著提高了[[FORMULA_1]]条件下材料性能。`

二つ目は数式があり、実際の組版負荷は大きいです。  
システムは `token_units(...)` で数式により高い視覚コストを与え、同じ需要とはみなしません。

## 六、第五層：visual line count を入れる理由

見落としがちな問題があります。

**OCR の行数が信用できないことがある**、という点です。

本来 4 行の段落が 1 行に結合されていると、生の `lines` だけでは可排空間を過大評価します。

[measurement.py](backend/scripts/services/rendering/layout/typography/measurement.py) にはこれを補正する関数群があります。

- `plain_text_chars_per_line(...)`
- `_predicted_wrapped_line_count(...)`
- `visual_line_count(...)`
- `is_tall_single_line_glue(...)`

`visual_line_count(...)` の考え方は次のとおりです。

- まず OCR の行数を見る
- テキスト長・箱幅・一行の載せ能力から「正常折り返しなら何行か」を推す
- 推定が OCR 行数を大きく上回れば、より保守的な行数を採用

目的はフォントサイズ計算そのものではなく、**密度判定が偽データに引っ張られないこと**です。

### 典型例

あるブロックで：

- OCR は 1 行のみ
- 箱は高いがテキストは 140 文字
- 幾何的に一行では収まり得ない

`visual_line_count(...)` は次のように判断します。

「これは 1 行の本文ではなく、OCR が複数行を 1 行に粘着した可能性が高い。」

予測値で以降の容量判定を補正し、密度が実態に近づきます。

---

## 七、密度が最終的にフォントサイズへ与える影響

これらの関数は直接「最終 pt 値」を返しません。組版器への判断材料です。

最も直接的な落点は [fit.py](backend/scripts/services/rendering/layout/payload/fit.py) の `fit_translated_block_metrics(...)` です。

```python
capacity = box_capacity_units(box, font_size_pt, leading_em, visual_lines=visual_lines)
if capacity <= 0 or (demand <= capacity * 0.96 and layout_density < LAYOUT_DENSITY_SAFE_MAX):
    return font_size_pt, leading_em
```

ここが重要です。

- 需要が容量に迫っていない
- かつレイアウト密度が高すぎない

なら現在のフォントサイズを維持できます。

逆に：

- `demand > capacity`
- または `layout_density` が高すぎる

ならフォントサイズ縮小・行送り圧縮のフローに入ります。

密度は「9.2pt」「8.8pt」を直接出力するのではなく、次を決めます。

- 縮小するか
- 何段階縮めるか
- フォントサイズのみか、行送りも含めるか

---

## 八、素朴な判断チェーンとして理解する

すべての関数を一文の鎖に圧縮すると、おおよそ次のとおりです。

1. **翻訳後コンテンツが明らかに長くなったか**
   - `translation_density_ratio(...)`

2. **現在のフォントサイズで、この箱に何行分の高さが要るか**
   - `layout_density_ratio(...)`

3. **より厳密に、箱が実際に載せられる単位量はいくつか**
   - `box_capacity_units(...)`

4. **OCR 行数を盲信せず、`visual_line_count(...)` で補正**

5. **需要と容量で縮小するか決める**
   - `text_demand_units(...)` vs `box_capacity_units(...)`

---

## 九、完全な例

ある翻訳ブロックに次の条件があるとします。

- 箱幅：145pt
- 箱高：62pt
- 初期フォントサイズ：9.4pt
- 行送り：0.58em
- 原文語数：18
- 翻訳後中国語文字：22
- 数式 2 個

システムの見方：

### ステップ 1：コンテンツの膨張

`translation_density_ratio = 22 / 18 ≈ 1.22`

重度コンパクトブロックです。

### ステップ 2：現在フォントでのレイアウト圧

`layout_density_ratio(...)` の概算では：

- 箱が狭い
- 9.4pt では一行の文字数が限られる
- 数式で実折り返し圧がさらに増える

レイアウト密度は `1.0` 前後、あるいは超える可能性があります。

### ステップ 3：容量と需要

- `box_capacity_units(...)` は総容量が小さいと判断
- `text_demand_units(...)` は数式で需要を押し上げる

結論：

**高密度ブロックで、現在のフォントサイズは安全ではない。**

### ステップ 4：収縮フェーズ

`fit_translated_block_metrics(...)` は次を試します。

- まずフォントサイズを段階的に下げる
- 足りなければ leading も少し圧す
- 需要が容量を明確に超えなくなるまで繰り返す

これが「密度判定がフォントサイズに効く」一連の流れです。

---

## 十、まとめ

いわゆる「箱の密度」が問うのは

「この箱に文字がいくつあるか」

ではなく、

**現在のフォントサイズと行送りにおいて、箱の利用可能容量と、翻訳コンテンツの実需要の差がどれだけか。**

現行実装では、次のモジュールが連携して答えます。

- ページと元行情報の補正：
  - [measurement.py](backend/scripts/services/rendering/layout/typography/measurement.py)
- テキスト長とレイアウト密度の概算：
  - [text_common.py](backend/scripts/services/rendering/layout/payload/text_common.py)
- 容量と需要の計算：
  - [capacity.py](backend/scripts/services/rendering/layout/payload/capacity.py)
- 最終的なフォントサイズ収縮の可否：
  - [fit.py](backend/scripts/services/rendering/layout/payload/fit.py)

「なぜ最終的に小さくなったか」を追うと、多くの場合次に帰着します。

**現在ブロックの内容需要が、現在のフォントサイズにおける箱の容量に迫った、あるいは超えたから。**
