# 工程評価と実行計画

本ドキュメントは、外部評価・現在のリポジトリ状況・最近の開発・デリバリー上の課題を踏まえた内部判断です。

README の繰り返しではなく、次の 3 点に答えます。

1. このプロジェクトは現時点でどの程度できているか
2. 本当の弱点は何か
3. 今後どの順序で進めるべきか

## 1. 現在のフェーズ

単なるスクリプト集やデモ段階を超えています。

既に次を備えています。

- ブラウザフロントエンド
- Rust API サービス層
- Python 処理パイプライン
- Docker デリバリー
- デスクトップパッケージング
- タスクディレクトリ、イベントストリーム、成果物ダウンロード、段階診断などのプロダクト化能力

より正確には:

- 「動くプロダクト原型」には到達
- 「低保守コストで安定デリバリーできる工程状態」には未到達

## 2. 外部評価で妥当な点

### 2.1 README をナビゲーションにする方針

1 ファイルですべてを説明するのはもはや困難です。README を総合索引にし、詳細を分散するのは正しい方向です。

### 2.2 フロント・バックエンド・パイプラインの形は揃っている

タスク投入・照会、OCR、翻訳、組版再構築、ダウンロード、Docker/デスクトップまで、プロダクトチェーンとしては一通り揃っています。

### 2.3 Rust API + Python pipeline は合理的

Rust がサーバー・スケジューリング・状態管理・インターフェース・デリバリーを担い、Python が OCR・翻訳・組版・実験を担う分担は本ユースケースに合います。

## 3. 外部評価より重要な、現在の核心課題

### 3.1 「単一の真源」がまだ硬くない

表面は CI 失敗、デスクトップビルド失敗、フロントのフィールド欠落、ダウンロードパス変更などに見えますが、本質は次の真源が未統一なことです。

- ドキュメント真源
- 設定真源
- 成果物真源
- デスクトップビルド入力真源
- パス・ディレクトリ構造真源

ローカルにあるが CI に無い、ローカル既定がクリーン環境で無い、ダウンロードパスがコード推論、ドキュメントにあるフィールドが API で安定返却されない、といったズレが起きます。機能を足すほど保守コストが増幅します。

### 3.2 最大の弱点はアルゴリズムより工程一貫性

コア価値は組版保持・翻訳品質・最終 PDF ですが、時間を消費しているのは多くが次です。

- 長ブロック翻訳の不安定
- 数式プレースホルダ保護不足
- ブロック失敗後の降格戦略不足
- 段階診断の分かりにくさ
- ローカルと CI のビルド差
- ドキュメントとコードの乖離

「より強いモデルを足す」より、「既存チェーンを安定・説明可能・排查しやすくする」方が先です。

### 3.3 リポジトリが大きいこと自体は問題ではない。境界が曖昧なことが問題

単一リポジトリ多コンポーネントは許容できます。問題は契約の硬度です。

- フロントが依存する安定 API
- バックエンドが長期約束するフィールド
- Python が受け取るパラメータ範囲
- ダウンロードがディレクトリ推論か DB 登録か
- デスクトップパッケージが依存する明示ファイル

境界が明確なら単一リポジトリでも保守可能です。

## 4. 「成熟」の判断基準

PDF が出れば成熟、ではなく次の 4 点を見ます。

### 4.1 クリーン環境で安定ビルドできるか

別マシンや GitHub Actions で、ローカル履歴ファイルや手作業リソース補完なしに再現可能か。最近の CI 連続失敗はここが未完了である示唆です。

### 4.2 API 契約は安定か

ドキュメント記載フィールドが安定返却されるか。タイムラインをフロントがイベントから再構成する必要がないか。成果物パスをバックエンドが統一公開しているか。

### 4.3 タスクと成果物は追跡可能か

何が起きたか、各段階の所要時間、失敗理由、最終ダウンロード対象が安定取得できるか。

### 4.4 翻訳チェーンに安定機構があるか

長段落が容易に壊れないか。数式・プレースホルダが失われにくいか。例外時に自動降格できるか。ユーザーに「タスク失敗」一行ではなく説明可能な失敗が見えるか。

## 5. 今後最も投資価値の高い方向

機能を漫然と増やすより、次の 3 本柱です。

### 5.1 第一本柱: デリバリー工程の収束

GitHub Actions の安定パッケージ、Windows デスクトップの再現ビルド、Docker/フロント/バックエンド材料の明確化、ローカル暗黙ファイル依存の排除。「自分のマシンで動く」から「他者のマシンでも安定デリバリー」へ。

### 5.2 第二本柱: バックエンド契約の硬化

API ドキュメントと実返却の一致、`runtime.stage_history` / `events` / `artifacts` の長期安定、ダウンロードの DB 登録化、リクエスト・永続化・下流透過の追跡可能性。

### 5.3 第三本柱: 翻訳安定性を「派手な機能」より優先

長文ブロック分割、数式・プレースホルダ保護、失敗リトライと降格、ページ単位品質診断の明確化。RAG や用語庫拡張より先。基盤が不安定だと拡張の価値が相殺されます。

## 6. 三段階の提案

### 段階一: 工程真源の統一

ドキュメント入口、バージョン真源、デスクトップ入力、設定来源、成果物登録方式の統一。

受け入れ: クリーン環境がローカル欠落ファイルで落ちない。ドキュメントの現行入口が一意に分かる。

### 段階二: API とタスク状態の安定契約

タスク詳細、段階タイムライン、イベント、成果物一覧、エラー診断構造の固定。

受け入れ: フロントがフィールドを推測しない。バックエンドが返すべき状態を自前再構成しない。

### 段階三: 翻訳品質安定性の集中改善

長ブロック戦略、数式保護、単ブロック失敗フォールバック、ページ品質診断、可観測性。

受け入れ: 同種 PDF の成功率向上。「失敗」が減り、失敗時も段階・ブロックが特定できる。

## 7. 一行結論

最強な点は単一アルゴリズムではなく、「PDF アップロード → OCR → 翻訳 → 組版再構築 → ダウンロード」チェーンを実際に通していることです。

今後最優先で磨くのは、工程一貫性、契約安定性、ビルド再現性、翻訳安定性です。これらが固まれば「強い原型」から「安定デリバリーできるプロダクト工程」へ進めます。
