# イベントストリーム API

## 1. エンドポイント

メインタスク:

`GET /api/v1/jobs/{job_id}/events?limit=50&offset=0`

OCR サブタスク:

`GET /api/v1/ocr/jobs/{job_id}/events?limit=50&offset=0`

## 2. 用途

主に次に使います。

- デバッグ
- 問題排查
- リトライチェーンの観察
- 各段階でバックエンドが何をしたかの確認

「過程タイムライン」の主真源ではありません。

- 主タイムライン: `runtime.stage_history`
- デバッグ明細: `/events`

## 3. 返却構造

トップレベル `items` ではなく:

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "job_id": "20260404150516-75857c",
        "seq": 1,
        "ts": "2026-04-04T15:05:16Z",
        "level": "info",
        "stage": "queued",
        "event": "job_created",
        "message": "タスクが作成されました",
        "payload": {}
      }
    ],
    "limit": 50,
    "offset": 0
  }
}
```

読み取りパス:

- `data.items`
- `data.limit`
- `data.offset`

## 4. 各イベントのフィールド

- `job_id`
- `seq`
- `ts`
- `level`
- `stage`
- `event`
- `message`
- `payload`

- `seq`: タスク内単調増加
- `level`: 通常 `info` / `warn` / `error`
- `stage`: 発生時の所属段階
- `event`: 機械可読イベント名
- `message`: 人間可読説明
- `payload`: 追加詳細

## 5. よくあるイベント種別

重点:

- `job_created`
- `status_changed`
- `stage_updated`
- `stage_transition`
- `stage_progress`
- `retry_scheduled`
- `failure_classified`
- `job_terminal`

互換で見えることもある:

- `job_error`
- `ocr_child_created`
- `ocr_child_finished`

## 6. デフォルトの永続化

イベントは通常、次の両方に書き込まれます。

- DB `events` テーブル
- `DATA_ROOT/jobs/{job_id}/logs/events.jsonl`

## 7. 画面が「イベント流れ空」になる原因

1. 包装層の読み取りミス（`data.items` をトップ `items` と誤認）
2. 別タスクの `job_id` をリクエスト
3. 極早期の履歴タスク（イベント永続化前）

最近の新タスクで `200` かつ `data.items` 非空なのに画面が空なら、ほぼ 1 です。

## 8. 返却例

タスク `20260404150516-75857c` は `200 OK` でイベントあり。API 自体は正常です。
