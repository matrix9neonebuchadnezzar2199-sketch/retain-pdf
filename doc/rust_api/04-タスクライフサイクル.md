# タスクライフサイクル

## 1. 現在サポートする workflow

1. `mineru` — OCR → Normalize → Translate → Render の完全チェーン
2. `ocr` — OCR/Normalize のみ、`document.v1.json` を出力
3. `translate` — OCR → Normalize → Translate（レンダリングなし）
4. `render` — 既存 job artifacts を再利用してレンダリングのみ再実行

## 2. メインチェーン

### 2.1 `mineru`

1. PDF アップロード
2. メインタスク作成
3. 内部で OCR サブタスク作成
4. OCR provider のアップロード・ポーリング・結果ダウンロード
5. `document.v1` へ標準化
6. 翻訳
7. レンダリング
8. PDF / ZIP 等を出力

### 2.2 `translate`

1. PDF アップロード
2. 翻訳タスク作成
3. 内部で OCR サブタスク
4. OCR 完了後に標準化・翻訳
5. translation payload、`translation-manifest.json`、診断情報を出力

補足:

- `translation.math_mode=placeholder` → 従来の公式保護チェーン
- `translation.math_mode=direct_typst` → 実験的「公式直出」分支
- タスク API とレンダ入口は変わらず、翻訳段階の公式処理のみ変化

### 2.3 `render`

1. レンダタスク作成
2. `source.artifact_job_id` で既存 job を参照
3. 当該 job の `source_pdf` と `translations_dir` を再利用
4. レンダリングのみ実行
5. 新しい PDF / Typst / レンダ成果物を出力

## 3. OCR サブタスクが存在する理由

`mineru` と `translate` は OCR をメインに直結せず、通常 `{job_id}-ocr` サブタスクを作ります。

- OCR transport と翻訳・レンダチェーンを分離観察できる
- provider 診断を OCR 側に単独で載せられる

## 4. よくある段階名

- `queued`
- `ocr_submitting`
- `ocr_upload`
- `mineru_upload`
- `mineru_processing`
- `translation_prepare`
- `normalizing`
- `translating`
- `domain_inference`
- `page_policies`
- `rendering`
- `finished`
- `failed`

補足:

- `translate` は一般に `translating` → `finished` で停止
- `render` は一般に `rendering` へ直行
- `ocr` は `translating` / `rendering` に入らない

## 5. リクエスト側の区別

1. `workflow` — 完全 / OCR-only / Translate-only / Render-only
2. `source`
   - `upload_id`: `mineru` / `translate`
   - `artifact_job_id`: `render`

- `workflow=translate` は OCR サブタスクを作るがレンダには入らない
- `workflow=render` は OCR・翻訳を再実行せず artifacts を再利用
- `workflow=ocr` は `/api/v1/ocr/jobs` 専用。`/api/v1/jobs` と混在させない
- `/api/v1/jobs` の JSON 正式契約は `source` / `ocr` / `translation` / `render` / `runtime` のグループ構造
- 履歴フラットフィールドは一部 multipart 補助入口のみ

## 6. フロントでの段階の理解

3 層に分ける:

- `status`: タスク終了したか
- `stage`: 現在どの段階か
- `stage_detail`: 人間向けの段階説明

例: `status=running`, `stage=translating`, `stage_detail=18/55 バッチ完了` → 未終了・翻訳中・18/55 まで進行。

## 7. ダウンロードボタンが有効になる条件

自前ルールを組まず、詳細 API の次を読む:

- `actions.*.enabled`
- `artifacts.*.ready`

- `translate` 成功時: 翻訳ディレクトリと bundle は ready になり得るが、PDF は必ずしも ready ではない
- `render` 成功時: PDF は ready になり得るが、OCR 関連ダウンロードは必ずしも無い

## 8. 失敗時に見るもの

優先:

- `failure.category`
- `failure.summary`
- `failure.root_cause`
- `failure.retryable`
- `failure.suggestion`

不足時:

- `runtime.final_failure_category`
- `runtime.final_failure_summary`
- `log_tail`
- `/events`

## 9. タイムラインとイベントの役割分担

- `runtime.stage_history`: 各段階の所要時間
- `/events`: 過程で起きた具体イベント
- `failure`: 失敗の分類・再試行可否・推奨対応
- `failure_diagnostic`: 旧フロント互換の簡易ビュー

概要用タイムライン、排查用イベント流れ。混用・相互逆算しない。
