# 成果物一覧とダウンロード

## 1. 成果物一覧が必要な理由

タスクディレクトリはバックエンド内部実装であり、将来変更され得ます。

フロントや外部呼び出しは次に依存すべきではありません。

- `rendered/` の固定構成
- `md/` の固定構成
- `ocr/unpacked/` の不変性

対外の正式な真源は DB 上の安定した成果物一覧であり、API で公開されます。

## 2. 主 API

メインタスク:

`GET /api/v1/jobs/{job_id}/artifacts-manifest`

OCR サブタスク:

`GET /api/v1/ocr/jobs/{job_id}/artifacts-manifest`

## 3. 返却内容

各エントリに少なくとも:

- `artifact_key`
- `artifact_group`
- `artifact_kind`
- `ready`
- `file_name`
- `content_type`
- `size_bytes`
- `relative_path`
- `checksum`
- `source_stage`
- `updated_at`
- `resource_path`
- `resource_url`

フロントが使うべきは `artifact_key`, `ready`, `resource_path` / `resource_url`。

詳細の `artifacts` はボタン状態向け。機械可読な完全探索は `artifacts-manifest` を優先。

## 4. よくある artifact_key

- `source_pdf` — `ocr.page_ranges` 指定時はタスク内の子セット PDF
- `translated_pdf`
- `typst_source`
- `typst_render_pdf`
- `markdown_raw`
- `markdown_images_dir`
- `markdown_bundle_zip`
- `normalized_document_json`
- `normalization_report_json`
- `layout_json`
- `translation_manifest_json`
- `provider_bundle_zip`
- `provider_result_json`
- `pipeline_summary`
- `events_jsonl` — 実行時 events.jsonl のエクスポート（デバッグ用）

## 5. 推奨読み取り

1. `artifacts-manifest` を取得
2. 目的の `artifact_key` を特定
3. `ready` を確認
4. `resource_path` または `resource_url` を使用

`markdown_bundle_zip` は Markdown 専用 ZIP（`markdown/full.md` と `markdown/images/**` のみ）。

`?include_job_dir=true` でルートを `{job_id}-markdown/` に変更可能。

イベントの主表示は `/events` を優先。`events_jsonl` はエクスポート・归档向け。

## 6. 旧ダウンロード URL との関係

- `/pdf`, `/markdown`, `/normalized-document` 等は残存
- 内部は「一覧 → 実ファイル」の方向に収束
- 詳細の `artifacts.pdf` 等が推奨。同級の `pdf_url` 等は互換エイリアス

旧タスクレイアウト・絶対パス artifact の場合、これらの入口も拒否され再実行が必要。

## 7. 安定性の理由

ディレクトリが `rendered/` から `outputs/render/` に変わっても、DB 登録を更新すれば外部 API は追従不要。公開契約から「ディレクトリ構造」を切り離すのが目的。
