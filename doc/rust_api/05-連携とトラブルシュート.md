# 連携とトラブルシュート

## 1. 最も多い 3 種の誤判断

### 1.1 バックエンド未返却か、フロント読み取りミスか

まず curl で生レスポンスを確認する。

```bash
curl -s http://127.0.0.1:41000/api/v1/jobs/<job_id> \
  -H 'X-API-Key: your-key'
```

```bash
curl -s 'http://127.0.0.1:41000/api/v1/jobs/<job_id>/events?limit=50&offset=0' \
  -H 'X-API-Key: your-key'
```

生レスポンスに値があるのに画面に無い → フロントの unwrap を疑う。

### 1.2 「runtime.stage_history が返らない」

2 パターン:

1. 履歴タスク: `runtime = null`（当時未永続化）
2. 新タスクで本当に未書き込み → 調査対象の異常

混同しない。

### 1.3 「詳細・ダウンロードが旧タスク非対応」

`runtime = null` とは別問題。

- `originPDF/jsonPDF/transPDF/typstPDF` 旧レイアウト
- 旧版絶対パス artifact 保存

→ 現行バックエンドは拒否し、再実行が必要。一時互換やフロント解析問題ではない。

### 1.4 「イベント流れが空」

1. HTTP `200` か
2. `data.items` 非空か
3. トップ `items` と誤読していないか

## 2. タスク詳細ページの推奨マッピング

- 現在状態: `data.status`
- 現在段階: `data.runtime.current_stage`、フォールバック `data.stage`
- 段階説明: `data.stage_detail`
- 過程タイムライン: `data.runtime.stage_history`
- 現在段階経過: `data.runtime.active_stage_elapsed_ms`
- 全体経過: `data.runtime.total_elapsed_ms`
- 失敗要約: `data.failure.summary`
- 失敗カテゴリ: `data.failure.category`
- 数式翻訳モード: `data.request_payload.translation.math_mode`
- 実行プロトコル: `data.invocation`
- 正規化要約: `data.normalization_summary`
- 用語集要約: `data.glossary_summary`
- ダウンロード: `data.actions.*.enabled`

## 3. イベント流れタブ

- 配列: `data.items`
- limit: `data.limit`
- offset: `data.offset`

各 item は `seq`, `ts`, `level`, `stage`, `event`, `message` を表示。`payload` は展開領域推奨。

## 4. ディスク上の確認

`DATA_ROOT/jobs/{job_id}/`

- `specs/`
- `logs/events.jsonl`
- `artifacts/pipeline_summary.json`
- `ocr/`, `translated/`, `rendered/`

失敗時の追加:

- `logs/failure-ai-diagnosis.request.json`
- `logs/failure-ai-diagnosis.response.json`

（主失敗カテゴリが `unknown` のとき AI 補助診断が走った場合のみ）

`specs/` は worker が実際に使う stage spec。無いのに新タスクと思ったら旧成果物や未完成ディレクトリを疑う。

## 5. 事例

タスク `20260404150516-75857c`:

- 詳細で `runtime = null` → 過程タイムラインは表示不可
- `/events` と `logs/events.jsonl` はデータあり

→ 実行時タイムライン永続化前の履歴タスク。イベント API は正常。

## 6. ドキュメントレベルの結論

- タイムライン: `runtime.stage_history` のみ
- イベント: `/events` のみ
- 実行プロトコル: `invocation`
- 正規化概観: `normalization_summary`
- 相互回填・混用・イベントからの主タイムライン逆算はしない

## 7. 失敗帰属の読み方

1. `data.failure.summary`
2. `data.failure.root_cause`
3. `data.failure.suggestion`
4. `data.failure.raw_diagnostic`
5. `data.failure.ai_diagnostic`

- `failure` が主真源
- `failure_diagnostic` は互換のみ

診断チェーン: Python 構造化失敗 JSON → Rust 分類 → `unknown` なら AI 補助 → 詳細・イベントへ永続化

## 8. 関連イベント

- `failure_classified`
- `failure_ai_diagnosed`（`unknown` のみ）
- `job_terminal`

## 9. OCR ページ範囲

```json
{
  "ocr": {
    "page_ranges": "1-5"
  }
}
```

- `request_payload` と DB `jobs.request_json` の `$.ocr.page_ranges` を確認
- `provider=mineru` かつ `source.upload_id` では子セット PDF を生成してアップロード
- 空文字列は全ページ

効いていれば `source_pdf`・翻訳ページ・最終 PDF は子セットのみのはず。

## 10. 旧タスク拒否時

1. ディレクトリが旧 4 分割か
2. artifact が絶対パスか
3. spec-only / 新 artifact レイアウト前のタスクか

→ 自動移行なし。再実行。

## 11. `math_mode` 連携

1. `request_payload.translation.math_mode`
2. `placeholder` → placeholder 系挙動は正常
3. `direct_typst` → 直出品質・英語残留・レンダ互換を重点確認
