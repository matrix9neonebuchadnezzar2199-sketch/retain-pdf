# タスク詳細とタイムライン

## 1. 主 API

タスク詳細ページの主 API:

`GET /api/v1/jobs/{job_id}`

詳細ページの大部分はこの 1 本から取得します。

- 現在の状態
- 現在の段階
- 段階の説明
- 実行時タイムライン
- 失敗の帰属
- ダウンロードボタンの状態
- OCR サブタスクの要約
- 実行プロトコルの要約
- 正規化要約と用語集要約

## 2. フロントが最優先で見るフィールド

重点フィールドは `data` 内:

- `request_payload`
- `status`
- `stage`
- `stage_detail`
- `runtime`
- `failure`
- `actions`
- `artifacts`
- `timestamps`
- `log_tail`
- `normalization_summary`
- `glossary_summary`
- `invocation`

補足:

- `request_payload` はバックエンドが当該タスク用に保存したリクエストパラメータのスナップショット
- 連携時にパラメータがバックエンドに入ったか確認する場合はここを優先
- OCR ページ範囲: `data.request_payload.ocr.page_ranges`
- 数式翻訳モード: `data.request_payload.translation.math_mode`
- `invocation` は旧経路か現在の stage spec 経路かを示す
- `normalization_summary` は `schema_version=1.1` か、デフォルト値収束が起きたかの要約
- `glossary_summary` は用語集の有効化・ヒット・未ヒットの要約

## 3. 新タスクの実行プロトコルの見方

新版タスクでは次が期待されます。

```json
{
  "invocation": {
    "stage": "mineru",
    "input_protocol": "stage_spec",
    "stage_spec_schema_version": "mineru.stage.v1"
  }
}
```

連携時の判断:

- `input_protocol=stage_spec` → 現在の spec-only worker 経路
- `stage_spec_schema_version` → 段階 spec のバージョン
- ここが無い場合、必ずしも API 障害ではなく、履歴タスクの可能性あり

## 4. タイムラインの真源

「概要 → 過程タイムライン」は必ず `runtime.stage_history` を真源とする。

`/events` から主タイムラインを逆算しない。理由:

- `runtime.stage_history` はバックエンドが整理済みの段階区間
- 各区間に入退場時刻がある
- 終端情報を直接付与できる
- フロントでマージ・重複除去・推測が不要

## 5. `runtime` の主要フィールド

- `current_stage`
- `stage_started_at`
- `last_stage_transition_at`
- `total_elapsed_ms`
- `active_stage_elapsed_ms`
- `retry_count`
- `last_retry_at`
- `terminal_reason`
- `final_failure_category`
- `final_failure_summary`
- `stage_history`

## 6. `stage_history` の固定構造

各項目は 1 つの段階時間片:

```json
{
  "stage": "translating",
  "detail": "翻訳中、12/22 バッチ完了",
  "enter_at": "2026-04-04T15:31:02Z",
  "exit_at": null,
  "duration_ms": null,
  "terminal_status": null
}
```

- `stage`: 段階名
- `detail`: 当該段階に入ったときの主説明
- `enter_at`: 入場時刻
- `exit_at`: 退場時刻。現在アクティブな段階は通常 null
- `duration_ms`: 段階完了時に安定。アクティブ段階は通常 null
- `terminal_status`: 終了直前の最後の段階なら `succeeded` / `failed` / `canceled`

## 7. 実行中タスクの読み方

実行中:

- 現在アクティブな段階も `stage_history` に含まれる
- 通常 `exit_at = null`、`duration_ms = null`
- 現在段階のリアルタイム経過は `runtime.active_stage_elapsed_ms` を読む

ローカル補間表示以外では、「現在時刻 − enter_at」を主表示に使わない。API 真源は `active_stage_elapsed_ms`。

## 8. 終了済みタスクの読み方

- `status` は `succeeded` / `failed` / `canceled`
- `runtime.terminal_reason` に終端理由
- `runtime.total_elapsed_ms` が全体所要時間
- `stage_history` 最後の項目に `terminal_status` が付くことが多い

## 9. `normalization_summary` の理解

タスク詳細の `normalization_summary` は軽量要約であり、完全レポートではない。

重点:

- `provider`
- `detected_provider`
- `schema`
- `schema_version`
- `document_defaults`
- `page_defaults`
- `block_defaults`
- `page_count`
- `block_count`

推奨:

- 画面表示はこの要約のみ
- adapter / defaults / validation の詳細排查は `artifacts.normalization_report` をダウンロード

注意:

- メインチェーンは `schema_version=1.1` のみ受理
- `*_defaults` はデフォルト値収束の件数（旧 `compat_*` ではない）

## 10. 履歴タスクでタイムラインが無い理由

履歴の古いタスクでは:

```json
{
  "runtime": null
}
```

これは次を意味しない:

- フロントの読み取りミス
- 現在のバックエンド障害
- 現在タスクの書き込み失敗

意味するのは:

- 作成・実行当時、実行時タイムラインが DB に永続化されていなかった

フロントは「履歴データ欠損」として扱う。

## 11. 「履歴欠損」と「旧タスク拒否」は別問題

- `runtime = null`: 履歴データ欠損。詳細・イベントは見えることがある
- 詳細/ダウンロードが「旧タスク非対応」: 旧ディレクトリレイアウトまたは旧 artifact 保存。再実行のみ

## 12. 保証範囲

`runtime.stage_history` がプロセス全体として保証されるのは:

- 新版バックエンドで作成されたタスク
- 同一新版バックエンドで最後まで実行されたタスク

## 13. フロント推奨読み取り順

1. `status`
2. `runtime.current_stage`
3. タイムラインは `runtime.stage_history`
4. 現在段階経過は `runtime.active_stage_elapsed_ms`
5. 全体経過は `runtime.total_elapsed_ms`
6. 失敗時は `failure.summary` と `failure.category`
7. 実行プロトコルは `invocation`
8. 正規化要約は `normalization_summary`
9. 用語集要約は `glossary_summary`

## 14. `translation.math_mode` の理解

- `placeholder`（デフォルト）: 従来の公式保護 / placeholder / 復元チェーン
- `direct_typst`（実験）: placeholder 保護なし。モデルが本文と `$...$` 数式を直接出力

フロントは文字列の表示・透過のみでよく、自前推測は不要。
