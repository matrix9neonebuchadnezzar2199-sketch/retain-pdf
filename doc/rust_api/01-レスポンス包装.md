# Rust API レスポンス包装

## 1. トップレベル形式

Rust API は現在、同一のレスポンス包装を使用します。

成功レスポンス:

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失敗レスポンス:

```json
{
  "code": 400,
  "message": "具体的なエラーメッセージ"
}
```

読み取りルール:

- `code = 0` は成功
- `message` は人間向け
- 業務データは常に `data` 内

## 2. 最も起きやすいミス

フロントの空白ページ・空リスト・「バックエンドにあるのに表示されない」多くは、API が返していないのではなく、読み取りパスが誤っていることが原因です。

典型的な誤り:

```json
{
  "items": [...]
}
```

これは現在の Rust API 構造ではありません。

正しい構造:

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [...]
  }
}
```

つまり:

- `response.items` は読まない
- HTTP クライアントの最外層にさらにオブジェクトがある場合、`response.data.items` もそのままでは読めない
- 最終的に API 返却体の `data` を読む

## 3. イベントストリーム API の具体例

`GET /api/v1/jobs/{job_id}/events?limit=50&offset=0`

実際の返却構造:

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "job_id": "20260404150516-75857c",
        "seq": 1,
        "ts": "2026-04-04T15:05:16Z",
        "level": "info",
        "stage": "queued",
        "event": "job_created",
        "message": "タスクが作成されました",
        "payload": {
          "stage": "queued",
          "status": "queued",
          "workflow": "mineru"
        }
      }
    ],
    "limit": 50,
    "offset": 0
  }
}
```

要点:

- イベント配列は `data.items`
- `limit` は `data.limit`
- `offset` は `data.offset`

## 4. 推奨するフロントエンドの unwrap 順序

タスク詳細・リスト・イベントストリームいずれも、次の順で処理することを推奨します。

1. HTTP ステータスコードを確認
2. レスポンスボディの `code` を確認
3. `code === 0` のときだけ `data` を読む
4. 業務フィールドをトップレベルに置くと仮定しない

## 5. フロントに直接書ける判定ロジック

- HTTP が `2xx` でない → ネットワークまたはサービスエラー
- HTTP が `2xx` だが `code !== 0` → 業務エラー
- `code === 0` かつ `data == null` → 成功だが業務ペイロードなし
- フィールドが `data.xxx` にあるべきなら、トップレベルで探さない
