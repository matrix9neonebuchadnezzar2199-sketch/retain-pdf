# 共同開発の約束

API 仕様書ではなく、`rust_api` を複数人で開発するときの境界約束です。

目的: 役割を踏みにくくし、構造を再び乱さないこと。

## 1. モジュール責務

- `routes/*` — HTTP adapter のみ（解析・認証後入口・レスポンス包装）。業務集約・job command 直組み立て禁止
- `services/jobs/*` — タスク域の業務（query, presentation, creation, control）。多くの業務変更はここ
- `services/job_factory.rs` — snapshot 組み立てと起動境界。「build snapshot」「start execution」の 2 段
- `job_runner/*` — 実行（キュー、キャンセル、プロセス、OCR 子タスク、render/translate チェーン）
- `models/*` — DTO と入出力・永続化モデルのみ。業務編成や FS 読み取りを増やさない

## 2. 依存ルール

- `routes -> services -> job_factory / job_runner / db`
- `job_runner -> db / config / runtime state`
- `models` は `routes` / `services` に依存しない

禁止:

- `routes` に複雑業務
- `models/view.rs` に FS 読み取り・失敗診断集約の復活
- 純粋 helper が `AppState` を直接消費

## 3. AppState

許可: ルート入口、ライフサイクル入口、キャンセル・実行スロット・プロセス協調が必要な実行モジュール。

狭依存化: コマンド構築、snapshot 組み立て、アップロード検証、読み取り view、DB/イベントのみの helper。

`&Db`, `&AppConfig`, `&Path`, `&RwLock<HashSet<String>>`, `&Arc<Semaphore>` で足りるなら `AppState` を渡さない。

## 4. 新要件の配置

- 照会フィールド・詳細表示・成果物 view → `services/jobs/presentation` または `query`
- 作成パラメータ・提出・bundle → `services/jobs/creation`
- 実行段階・子タスク・キャンセル → `job_runner/*`
- 共通入出力フィールド → `models/*`
- HTTP 入出力のみ → `routes/*`（業務判断は route に残さない）

## 5. 反パターン

- route で DB・view・command・FS を一手に
- `services` に HTTP ヘッダ・Multipart を直接記述
- `job_runner` に新しい `AppState` 透過 helper
- `models` でディスク読み取り集約
- 複数責務の単一ブラックボックス関数への再集約

## 6. 提出前の最低確認

- `cargo check --quiet`
- creation / job_factory 変更 → 対応 service テスト
- process runner / lifecycle 変更 → runner テスト
- query / presentation 変更 → 対応テスト

下位 helper が再び `AppState` に依存する場合はレビューで理由を説明すること。
